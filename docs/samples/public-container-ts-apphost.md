# Public Container TypeScript AppHost Sample (`@sekiban/aspire`)

The [`Sekiban.Dcb.WasmRuntime.PublicContainer.TsAspire`](../../src/samples/Sekiban.Dcb.WasmRuntime.PublicContainer.TsAspire)
sample hosts the **public GHCR runtime container** from an **Aspire 13.x
TypeScript AppHost** (`apphost.mts`), using the thin `@sekiban/aspire` npm
helper (`src/lib/sekiban-aspire-ts`). It is the TypeScript counterpart of the
C# `PublicContainer.CsDecider` AppHost.

What it demonstrates:

- An `apphost.mts` created the official way (`aspire new aspire-ts-empty`
  layout: `aspire.config.json`, `tsconfig.apphost.json`, generated
  `.aspire/modules` — regenerated by `aspire run`, not committed) with the
  PostgreSQL integration recorded in `aspire.config.json`.
- One `addSekibanWasmRuntime(builder, "runtime", {...})` call registering
  `ghcr.io/j-tech-japan/sekiban-wasm-runtime-host:1.0.0-preview.3`
  (`SAMPLE_RUNTIME_IMAGE_TAG` override honored) with read-only
  `/app/config` + `/app/modules` bind mounts, the runtime env contract, the
  `SekibanDcb` / `DcbMaterializedViewPostgres` references, and the HTTP
  endpoint.
- A smoke that consumes the **packed `@sekiban/aspire` tarball** (installed
  with `--no-save` over the dev-time `file:` reference), launches the AppHost
  headlessly via `aspire run`, and proves health/ready plus a command/query
  round trip (commit `WeatherForecastCreated` → `tag-latest-sortable`
  readback → `GetWeatherForecastListQuery`).
- The wasm module + manifest are shared with the CsDecider sample
  (`bash src/samples/Sekiban.Dcb.WasmRuntime.PublicContainer.CsDecider/scripts/build-wasm.sh`)
  — the hosting model is language-agnostic, so the same module runs no matter
  which AppHost language starts the container.

Run it:

```bash
bash src/samples/Sekiban.Dcb.WasmRuntime.PublicContainer.CsDecider/scripts/build-wasm.sh
bash src/samples/Sekiban.Dcb.WasmRuntime.PublicContainer.TsAspire/scripts/smoke.sh
```

Prerequisites: Docker, Node.js 20+, and the Aspire CLI 13.4+ (`aspire run`
drives TypeScript AppHosts).

## TS helper vs C# package — which to use

| | `@sekiban/aspire` (npm) | `Sekiban.Dcb.WasmRuntime.Aspire` (NuGet) |
| --- | --- | --- |
| AppHost language | TypeScript (`apphost.mts`) | C# (`AppHost.cs`) |
| Shape | Thin function `addSekibanWasmRuntime(builder, name, config)` over the official Aspire TS AppHost generated API; structural types, no validation layer | `AddSekibanWasmRuntime(name, options)` extension with a typed `SekibanWasmRuntimeOptions` class |
| Surface | Sample-driven minimum: image/tag (env-var override), config/modules mounts, references, env overrides, host port | The accepted fixed options set including manifest path, projection mode, endpoint naming, health-check path |
| Distribution | npm `@sekiban/aspire` 0.1.0 via the `ts-v*` lane | NuGet via the GitHub Release + `nuget-preview` lane |

Both wire the identical container contract (same image, mounts, env vars,
database names) and deliberately add no Postgres `WaitFor` gate — the runtime
connects lazily and retries. Choose by AppHost language; nothing about the
runtime differs. The TS helper intentionally does **not** clone the C# options
abstraction (grill decision 9): if you outgrow the helper, keep chaining the
official Aspire TS APIs on its return value.

See the package README (`src/lib/sekiban-aspire-ts/README.md`) for the helper
API, and [`public-container-cs-decider.md`](./public-container-cs-decider.md)
for the C# counterpart.
